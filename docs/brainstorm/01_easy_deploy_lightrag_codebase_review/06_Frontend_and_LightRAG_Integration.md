| `searchLabels(query, limit)` | `GET /api/lightrag/domains/{port}/graph/label/search` |

The frontend selects the LightRAG domain port from `client/src/stores/lightrag-domain-store.ts`.

## 6.3 Frontend Auth Store

File: `client/src/stores/auth-store.ts`

Responsibilities:

- Maintains auth state using `useSyncExternalStore`.
- Calls `authApi.me`, `authApi.login`, `authApi.logout`, and `authApi.changePassword`.
- Tracks status values such as `idle`, `loading`, `authenticated`, and `unauthenticated`.
- Exposes selectors such as `selectIsAdmin` and `selectCanWrite`.
- Redirects to `/login` after logout.

## 6.4 Frontend Chat Session Store

File: `client/src/stores/chat-session-store.ts`

Responsibilities:

- Tracks current conversation ID.
- Stores messages and assistant progress.
- Stores context payloads by assistant message ID.
- Maintains source tree state.
- Merges source trees across events while preserving unique child/frame IDs.

This store aligns with the backend chat stream events generated by `src/server/context/stream_service.py`.

## 6.5 Frontend Domain Store

File: `client/src/stores/lightrag-domain-store.ts`

Responsibilities:

- Stores selected LightRAG port.
- Defaults selected port to `9621`.
- Calls `fetchLightRagDomains` to load backend-discovered domains.
- Preserves current port if still available; otherwise picks the first available domain.

This means the frontend domain dropdown/selection is only as accurate as the backend domain discovery state.

## 6.7 LightRAG Runtime Integration

The repo embeds a customized LightRAG implementation under `src/lightrag`.

Important observed paths:

| Path | Purpose |
|---|---|
| `src/lightrag/lightrag.py` | Core LightRAG implementation entry area. |
| `src/lightrag/api/lightrag_server.py` | Native LightRAG API server. |
| `src/lightrag/api/routers/document_routes.py` | Native document APIs. |
| `src/lightrag/api/routers/query_routes.py` | Native query/context APIs. |
| `src/lightrag/api/routers/graph_routes.py` | Native graph APIs. |
| `src/lightrag/api/routers/ollama_api.py` | Ollama-compatible API behavior. |
| `src/lightrag/structural_chunking.py` | Custom structural chunking/artifact creation. |

The `Dockerfile.lightrag-local` copies `src/lightrag` into `/app/lightrag`, so local code changes in `src/lightrag` affect the LightRAG domain container image.

## 6.8 LightRAG WebUI

The repo also contains `src/lightrag_webui`, a separate React-based web interface for the LightRAG API server.

This is separate from the custom Next.js client in `client/`.

Potential implication:

```text
There are two frontend/UI concepts:
   1. Custom Next.js app under client/
   2. LightRAG WebUI under src/lightrag_webui/
```

Before removing or modifying either UI, confirm which one is actively used by operators/users.

## 6.10 Structural Chunking Integration

File: `src/lightrag/structural_chunking.py`

Important behavior:

- Detects structural-capable extensions.
- Locates per-domain root for source files.
- Builds artifact paths per document.
- Uses Docling-derived structures when available.
- Produces manifests with sections, blocks, assets, page counts, doc IDs, and raw Docling JSON paths.
- Provides metadata back to retrieval/source-tree features.

This is important because the custom frontend/source-tree UI appears to depend on rich metadata such as:

- page start/end
- section path
- chunk type
- artifact manifest path
- block IDs
- Docling references
- artifact references
- source type

## 6.11 Integration Risks

| Risk | Why It Matters |
|---|---|
| Two UI layers | Developers may update the wrong UI. Confirm whether `client/` or `src/lightrag_webui/` is the intended long-term interface. |
| Localhost LightRAG port override | Backend builds `http://127.0.0.1:{port}` for selected domain. This is fine for local host deployment, but can break in containerized backend deployment. |
| Vendored LightRAG code | Upstream changes and local modifications can diverge. Track what is custom. |
| Domain discovery at backend startup | If domains start after backend, UI may not see them unless refresh/discovery is re-run. |
| Graph write permissions | Frontend should hide/disable graph edit controls for users lacking `admin` or `can_write`; backend enforces this. |
