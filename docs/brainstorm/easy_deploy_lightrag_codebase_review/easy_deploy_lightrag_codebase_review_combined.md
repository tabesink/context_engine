# 1. Executive Summary and Architecture Map

## 1.1 What This Codebase Does

`easy-deploy-lightrag` is a local deployment and management wrapper around LightRAG domain-specific knowledge bases. It also includes a custom minimal backend and frontend for multi-user chat, graph visualization, and admin/user management.

The repo is best understood as a combination of:

1. **Deployment/orchestration layer** — `cli/main.py`, `docker-compose.domains.yml`, `Dockerfile.lightrag-local`, generated `data/*.env`, and per-domain folders.
2. **Minimal FastAPI backend** — `src/server/app/main.py`, auth, users, conversations, LightRAG graph proxy, SQLite persistence.
3. **Next.js management/chat client** — `client/src/app`, `client/src/stores`, `client/src/api/lightrag.ts`.
4. **Vendored/modified LightRAG runtime** — `src/lightrag`, including LightRAG API routers and structural chunking customization.
5. **LightRAG WebUI** — `src/lightrag_webui`, separate from the custom `client/` app.

## 1.2 Intended Users

| User Type | Likely Responsibilities |
|---|---|
| Operator / Admin | Run setup, deploy domains, start/stop Docker services, remove/recreate domains, manage users, edit graph entities/relations. |
| Normal user | Log in, chat/query against a LightRAG domain, view retrieved context/source tree, possibly view graph depending on UI permissions. |
| Developer / Coding agent | Maintain CLI, backend, client, Docker config, LightRAG integration, tests, and docs. |

## 1.3 Main Runtime Components

| Component | Main Paths | Purpose |
|---|---|---|
| CLI | `cli/main.py`, `cli/chat.py` | Typer/Rich command surface for setup, domain lifecycle, status, chat/TUI. |
| FastAPI backend | `src/server/app/main.py` | Auth, users, conversations, chat streaming, LightRAG graph proxy. |
| Auth/user layer | `src/server/auth/*`, `src/server/users/*` | JWT cookie auth, admin bootstrap, user CRUD, password changes. |
| SQLite storage | `src/server/storage/db.py`, `src/server/storage/repositories/*` | Backend persistence for users, audit log, conversations, messages, snapshots, rate-limit events. |
| Docker Compose runtime | `docker-compose.domains.yml`, generated by `cli/main.py` | Runs PostgreSQL/pgvector, Redis, Neo4j, and one or more LightRAG domain containers. |
| LightRAG domain container | `Dockerfile.lightrag-local`, `src/lightrag` | Runs LightRAG API/server with domain workspace, input dir, working dir, graph/vector stores. |
| Next.js client | `client/src/*` | Login, chat, settings/users, database/graph visualization. |
| LightRAG WebUI | `src/lightrag_webui/*` | Upstream-style React web UI embedded with LightRAG API server. |

## 1.5 What This Repo Appears to Handle

- Creates runtime config files and secrets under `data/`.
- Generates and updates `docker-compose.domains.yml`.
- Adds/removes per-domain LightRAG services.
- Runs shared PostgreSQL, Redis, and Neo4j services.
- Runs LightRAG domain containers using a customized LightRAG image build.
- Provides a backend auth/session/chat/graph proxy layer.
- Provides a custom Next.js client for chat, user settings, and graph visualization.
- Provides structural document chunking/custom artifact generation in `src/lightrag/structural_chunking.py`.

## 1.6 What This Repo Does Not Clearly Handle Yet

The following areas were not conclusively verified from visible files:

- Full native LightRAG document upload/index API behavior. The native route files exist under `src/lightrag/api/routers/`, but not every endpoint was enumerated in this review.
- A complete production hardening model for internet-facing deployment.
- A single consolidated backend namespace: visible runtime code uses `src/server`, while `pyproject.toml` and several tests reference `backend`.
- A root `README.md`: `pyproject.toml` references it, but public raw access returned 404 during this review.

### Chat Stream Event Flow

Implemented in `src/server/context/stream_service.py`, function `run_chat_turn()`:

```text
User POST /api/conversations/{id}/messages
   │
   ├── create user message
   ├── create placeholder assistant message
   ├── yield metadata event
   ├── yield progress: preparing
   ├── retrieve context unless selected_context provided
   ├── yield context event
   ├── save source tree snapshot
   ├── call LightRAG /query
   ├── yield token events
   ├── mark assistant message completed
   └── yield done event
```

On exception:

```text
mark assistant message failed
yield error event
yield done event
```

Primary CLI file: `cli/main.py`  
Interactive chat file: `cli/chat.py`

## 3.5 Dockerfile Behavior

File: `Dockerfile.lightrag-local`

Key behavior:

1. Uses upstream LightRAG image: `ghcr.io/hkuds/lightrag:latest`.
2. Installs extra Python/system dependencies, including `docling`, `pyuca`, `libxcb1`, `libgl1`, `libglib2.0-0`.
3. Copies local `src/lightrag` into `/app/lightrag`, overriding/extending the LightRAG code in the image.

Implication: this repo is not only orchestrating LightRAG; it also injects a local customized LightRAG implementation into the runtime image.

# 4. Data Model and Persistence

## 4.1 Backend Database

The custom FastAPI backend uses SQLite through `src/server/storage/db.py`.

`Settings.database_url` supports:

- `sqlite:///:memory:`
- `sqlite:///path/to/file.sqlite3`

Other database URL schemes are rejected by `database_path_from_url()`.

## 4.2 SQLite Tables

Schema is defined in `src/server/storage/db.py` as `SCHEMA_SQL`.

| Table | Purpose | Important Fields |
|---|---|---|
| `users` | User accounts and roles. | `id`, `username`, `password_hash`, `role`, `can_write`, `settings_visited`, `created_at`, `updated_at`, `last_login_at` |
| `audit_log` | Security/admin audit events. | `id`, `user_id`, `action`, `target_type`, `target_id`, `metadata`, `created_at` |
| `conversations` | Chat conversation metadata. | `id`, `user_id`, `title`, `last_message_preview`, `created_at`, `updated_at` |
| `messages` | Chat messages. | `id`, `conversation_id`, `role`, `content`, `status`, `sequence_no`, `metadata`, `created_at`, `updated_at` |
| `source_tree_snapshots` | Latest source tree/context snapshot per conversation. | `conversation_id`, `tree_json`, `updated_at` |
| `rate_limit_events` | Chat rate-limit events. | `id`, `user_id`, `bucket_start`, `created_at` |

Foreign keys are enabled with `PRAGMA foreign_keys = ON`.

## 4.4 Backend Database Flow

```text
FastAPI dependency
   │
   ▼
get_db(settings)
   │
   ├── database_path_from_url(settings.database_url)
   ├── sqlite3.connect(...)
   ├── PRAGMA foreign_keys = ON
   ├── executes SCHEMA_SQL
   └── yields connection through context manager
```

Important implication: schema creation appears to be application-managed at connection time. No Alembic migration flow was observed in the visible `src/server` backend.

## 4.5 Domain Runtime Storage

The CLI creates and manages domain-specific storage under `data/domains/<domain>/`.

Expected per-domain layout:

```text
data/domains/<domain>/
   ├── .env
   ├── inputs/
   ├── rag_storage/
   └── artifacts/
```

| Location | Created / Used By | Purpose | Safe to Delete? |
|---|---|---|---|
