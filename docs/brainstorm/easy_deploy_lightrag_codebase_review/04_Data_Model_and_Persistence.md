# 4. Data Model and Persistence

## 4.1 Backend Database

The custom FastAPI backend uses SQLite through `src/server/storage/db.py`.

`Settings.database_url` supports:

- `sqlite:///:memory:`
- `sqlite:///path/to/file.sqlite3`

Other database URL schemes are rejected by `database_path_from_url()`.

## 4.2 SQLite Tables

Schema is defined in `src/server/storage/db.py` as `SCHEMA_SQL`.

| Table | Purpose | Important Fields |
|---|---|---|
| `users` | User accounts and roles. | `id`, `username`, `password_hash`, `role`, `can_write`, `settings_visited`, `created_at`, `updated_at`, `last_login_at` |
| `audit_log` | Security/admin audit events. | `id`, `user_id`, `action`, `target_type`, `target_id`, `metadata`, `created_at` |
| `conversations` | Chat conversation metadata. | `id`, `user_id`, `title`, `last_message_preview`, `created_at`, `updated_at` |
| `messages` | Chat messages. | `id`, `conversation_id`, `role`, `content`, `status`, `sequence_no`, `metadata`, `created_at`, `updated_at` |
| `source_tree_snapshots` | Latest source tree/context snapshot per conversation. | `conversation_id`, `tree_json`, `updated_at` |
| `rate_limit_events` | Chat rate-limit events. | `id`, `user_id`, `bucket_start`, `created_at` |

Foreign keys are enabled with `PRAGMA foreign_keys = ON`.

## 4.4 Backend Database Flow

```text
FastAPI dependency
   │
   ▼
get_db(settings)
   │
   ├── database_path_from_url(settings.database_url)
   ├── sqlite3.connect(...)
   ├── PRAGMA foreign_keys = ON
   ├── executes SCHEMA_SQL
   └── yields connection through context manager
```

Important implication: schema creation appears to be application-managed at connection time. No Alembic migration flow was observed in the visible `src/server` backend.

## 4.5 Domain Runtime Storage

The CLI creates and manages domain-specific storage under `data/domains/<domain>/`.

Expected per-domain layout:

```text
data/domains/<domain>/
   ├── .env
   ├── inputs/
   ├── rag_storage/
   └── artifacts/
```

| Location | Created / Used By | Purpose | Safe to Delete? |
|---|---|---|---|
| `data/domains/<domain>/.env` | `cli/main.py` / `_write_domain_env()` | Domain-specific port/workspace/path config. | No, unless regenerating domain config intentionally. |
| `data/domains/<domain>/inputs/` | CLI + LightRAG container | Source/input documents. | Dangerous; may remove source corpus. |
| `data/domains/<domain>/rag_storage/` | LightRAG | Working/index storage mounted into container. | Dangerous; may force rebuild/reindex. |
| `data/domains/<domain>/artifacts/` | Structural chunking / Docling pipeline | Derived manifests, sections, pages, extracted assets. | Dangerous if UI/source-tree relies on artifacts. |
| `data/domains/_deleted/` | `_remove_domain()` | Archived removed domains. | Review before cleanup. |

## 4.7 Runtime Env and Secrets Storage

The CLI writes several env/config files under `data/`.

| File | Purpose | Risk |
|---|---|---|
| `data/base.env` | Shared LightRAG runtime defaults. | Model/storage changes affect all domains. |
| `data/postgres.env` | PostgreSQL credentials. | Secret; do not commit. |
| `data/redis.env` | Redis password. | Secret; do not commit. |
| `data/neo4j.env` | Neo4j connection values for LightRAG containers. | Secret-ish; contains password/URI. |
| `data/neo4j-server.env` | Neo4j server credentials. | Secret; do not commit. |
| `data/domains.json` | Domain manifest used by CLI/backend. | Controls available domains and ports. |

`cli/main.py` generates secrets using `secrets.token_urlsafe(32)` and applies restrictive permissions when possible.

## 4.8 Structural Artifacts

File: `src/lightrag/structural_chunking.py`

Important functions observed:

| Function | Purpose |
|---|---|
| `is_docling_structural_extension()` | Detects structural/docling-capable file extensions. |
| `domain_root_for_file()` | Locates the relevant `data/domains/<domain>` root for a source file. |
| `artifact_paths_for_doc()` | Computes artifact paths for a document. |
| `build_structural_artifacts()` | Builds manifest, blocks, sections, assets, raw Docling JSON, and metadata. |

Artifact output appears to include:

```text
data/domains/<domain>/artifacts/<doc_id>/
   ├── manifest.json
   └── docling_document.json   # when available
```

The manifest contains metadata such as:

- `version`
- `doc_id`
- `source_file`
- `source_engine`
- `content_rendering`
- `status`
- `warnings`
- `blocks`
- `sections`
- `assets`
- `raw_docling_path`

## 4.9 Source Tree Persistence

Source tree snapshots are produced in `src/server/context/workspace_tree.py` and saved by `src/server/context/stream_service.py` through `source_tree_repo.upsert_source_tree()`.

Purpose:

- Preserve the retrieved context tree shown to the UI.
- Map chunks back to document/page/section/artifact handles.
- Support later display of source evidence from a conversation.

## 4.10 Persistence Risks

| Risk | Evidence / Reason |
|---|---|
| Backend DB path mismatch | `.env.example` references `data/backend/backend.sqlite3`, while backend settings default to `data/server/server.sqlite3`. |
| No migration framework observed for `src/server` SQLite schema | Schema is created from `SCHEMA_SQL`. Future schema changes need a deliberate migration strategy. |
| Rate-limit table growth | `rate_limit_events` records events; no pruning behavior was observed in the reviewed repository code. |
| Domain remove is destructive | `_remove_domain()` removes active domain folder after archiving. Operators must understand archive location and recovery. |
| Shared service storage affects multiple domains | PostgreSQL/Redis/Neo4j are shared. Cleanup/reset can affect all domains. |
